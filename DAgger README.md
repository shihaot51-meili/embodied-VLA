## 采集 DAgger 数据对训练的影响

### 1. 什么是 DAgger

**DAgger**（Dataset Aggregation，数据集聚合）是一种交互式模仿学习方法，核心思想是：**让模型自己跑，人在模型犯错时提供纠正动作**。

```
普通模仿学习（Behavior Cloning）:
  人类示教 → 训练模型 → 部署
  问题: 模型犯错后进入从未见过的状态 → 不知道怎么恢复 → 错误累积

DAgger:
  人类示教 → 训练模型 → 模型执行 → 人纠正错误 → 补充数据 → 重新训练
              ↑                                    │
              └────────────────────────────────────┘
                    迭代循环
```

### 2. DAgger 数据 vs 普通示教数据的区别

| 维度 | 普通示教数据 | DAgger 数据 |
|------|------------|------------|
| **数据来源** | 人类全程操作 | 模型执行 + 人类纠正 |
| **状态分布** | 专家轨迹（理想状态） | 模型实际到达的状态（含错误状态） |
| **动作标签** | 人类动作 | 人类纠正动作 |
| **覆盖范围** | 只覆盖"正确路径" | 覆盖"正确路径 + 错误恢复路径" |
| **核心价值** | 学会做任务 | 学会从错误中恢复 |

```
普通示教数据:
  状态A → 状态B → 状态C → 状态D → 成功
  ↑ 只覆盖理想路径

DAgger 数据:
  状态A → 状态B → 状态X(偏离) → 状态Y(更偏) → 人纠正 → 状态C → 状态D → 成功
  ↑ 包含模型犯错后的状态和恢复动作
```

### 3. 对训练的积极影响

#### 3.1 解决分布偏移问题（Covariate Shift）

```
普通模仿学习的致命问题:
  训练数据: 人类总是从理想状态操作
  推理时: 模型犯一个小错 → 进入训练数据中从未见过的状态 → 不知道怎么办 → 犯更大的错 → 错误累积

DAgger 的解决:
  DAgger 数据包含模型犯错后的状态
  → 模型见过"偏离状态"并学过"如何恢复"
  → 推理时犯错也能自我纠正
  → 错误不再累积
```

#### 3.2 提升鲁棒性

| 方面 | 效果 |
|------|------|
| **错误恢复能力** | 模型学会从偏移状态回到正确轨迹 |
| **泛化能力** | 见过更多样的状态分布，适应力更强 |
| **实际部署成功率** | 大幅提升（因为能处理"没见过的情况"） |

#### 3.3 补充长尾场景

```
普通示教:
  人类操作熟练 → 轨迹都很"标准"
  → 罕见/困难场景数据少

DAgger:
  模型操作不熟练 → 会遇到各种意外情况
  → 人纠正这些意外 → 补充了长尾场景数据
```

### 4. 对训练的潜在挑战

#### 4.1 数据质量混合

```
数据集 = 原始示教数据 + DAgger 数据

潜在问题:
  - DAgger 数据中模型犯错时的状态可能"不自然"
  - 人类纠正动作可能不够流畅（反应延迟、操作不精确）
  - 两种数据的质量和分布不一致
```

**应对**：
- DAgger 数据需要和普通数据一样经过前处理（异常值滤除、动作对齐）
- 可能需要给不同来源的数据设置不同权重

#### 4.2 对 norm 统计量的影响

```
普通数据 norm:
  state 分布 = 专家轨迹的状态分布
  mean/std 反映"理想操作"的统计特征

加入 DAgger 数据后:
  state 分布 = 专家轨迹 + 偏移状态
  mean/std 范围可能变大（偏移状态值更极端）
  → 需要重新计算 norm_stats.json
  → 训练和推理必须用新的 norm
```

#### 4.3 对 config 的影响

| config 模块 | 影响 | 需要修改 |
|------------|------|---------|
| `data.repo_id` | 数据集路径变化（合并了 DAgger 数据） | ✅ 指向新合并数据集 |
| `assets` | norm 需要重新计算 | ✅ 指向新 norm |
| `num_train_steps` | 数据量增加，可能需要更多步数 | 可能需要增加 |
| `batch_size` | 数据多样性增加，batch 可适当增大 | 可选 |
| `lr_schedule` | 一般不变 | ❌ 保持 |
| `model` | 一般不变 | ❌ 保持 |

#### 4.4 训练动态变化

```
加入 DAgger 数据后训练 loss 曲线可能的变化:

原始数据训练:
  loss: 2.0 ──→ 0.15（平滑下降）

加入 DAgger 数据后:
  loss: 2.0 ──→ 0.18 ──→ 0.12
              ↑ 初始收敛稍高（数据更复杂）
                ↑ 最终 loss 可能更低（学到更多模式）
                
  也可能出现:
  loss: 2.0 ──→ 0.15 ──→ 0.20 ──→ 0.10
              ↑ 原始收敛    ↑ 加入DAgger后loss上升    ↑ 重新收敛到更低
              （因为新数据引入了新的、更难的状态）
```

### 5. DAgger 数据采集流程（融入现有流水线）

```
┌─ 第一轮 ──────────────────────────────────────────┐
│ 阶段1: 前处理原始示教数据                          │
│ 阶段2: 合并数据                                    │
│ 阶段3: 编写 config                                 │
│ 阶段4: 计算 norm                                   │
│ 阶段5: 训练 → 得到 checkpoint_v1                   │
└───────────────────────────────────────────────────┘
                    │
                    ▼
┌─ DAgger 数据采集 ─────────────────────────────────┐
│ 1. 部署 checkpoint_v1 到机器人                     │
│ 2. 模型自主执行任务                                │
│ 3. 模型犯错时，人类遥操作提供纠正动作              │
│ 4. 记录 [模型状态, 人类纠正动作] 对                 │
│ 5. 收集 N 条 DAgger 数据                           │
└───────────────────────────────────────────────────┘
                    │
                    ▼
┌─ 第二轮 ──────────────────────────────────────────┐
│ 阶段1: 前处理 DAgger 数据（同原始数据流程）         │
│ 阶段2: 合并 [原始数据 + DAgger 数据]               │
│ 阶段3: 修改 config（from ckpt，指向 checkpoint_v1）│
│        num_train_steps=20000, keep_period=1000    │
│ 阶段4: 重新计算 norm（数据分布变了）                │
│ 阶段5: 训练 → 得到 checkpoint_v2                   │
└───────────────────────────────────────────────────┘
```

### 6. DAgger 数据量与效果的关系

| DAgger 数据占比 | 效果 | 风险 |
|----------------|------|------|
| 0% | 基线，错误累积问题存在 | 推理时犯错不可恢复 |
| 10~20% | 最佳比例，补充恢复能力 | 风险低 |
| 30~50% | 可能引入过多"错误状态" | 数据质量下降 |
| >50% | 数据被错误状态主导 | 模型可能学到"不自然"的行为 |

**建议**：DAgger 数据通常占总量 10~20% 效果最好，既补充了恢复能力，又不破坏原始数据的质量。

### 7. 对 RTC 的影响

```
DAgger 数据的特殊性:
  - 模型执行时有推理延迟
  - 人类纠正时也有反应延迟
  → DAgger 数据天然包含延迟场景

对 RTC 训练的影响:
  - 正面: DAgger 数据中的延迟场景与 RTC 模拟的延迟一致
  → RTC 训练效果可能更好（数据更真实）
  - 注意: DAgger 数据的延迟分布可能与 bimodal 采样不完全一致
  → 可能需要调整 RTC 参数
```

### 8. 一句话总结

采集 DAgger 数据能显著提升模型的**错误恢复能力和实际部署成功率**，核心价值是让模型见过"犯错后的状态"并学会纠正。但需要注意：DAgger 数据需经过同样的前处理流程、需要重新计算 norm、config 需改为 from ckpt 模式（2万步），建议 DAgger 数据占比 10~20% 以平衡恢复能力和数据质量。在现有流水线中，DAgger 是一个"训练→部署→采集→再训练"的迭代过程，每轮都能让模型更鲁棒。
